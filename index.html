03 / 核心痛點與修正狀態

承接意圖與避免重複追問

① 制式追問 已修正

「我想下週來做電音波」舊回覆只問「哪個部位」。已新增 treatment_booking_ready：療程+時間+來做 → 直接進 booking-aware 接待，電音波 保留為獨立標籤。

② RAG 重複問同題 已修正

客戶答完探查題後 query rewrite 又問同題。已新增 _build_consultation_answer_followup() 與 _is_nasolabial_severity_reply() guard，禁止重問笑/不笑。

04 / 更新後優化方案

策略一～三

一：以 FacebookAgent 為主體

短期優化落在 _detect/_handle_contextual_booking_request()、_should_explore_first()、_handle_standard_rag()、RiskGuard、booking_flow。Multi-Agent 為中長期。

二：RAG 治理 + Hybrid Search

每次更新產生 ingestion manifest；高頻療程建 query set（電音波、音波、電波、法令紋、皮秒、肉毒、玻尿酸），用 debug_rag.py 驗證 retrieval。

三：價格 no-quote 更像顧問

同理 → 說明費用變因 → 問不造成負擔的狀態題 → 邀請專人/諮詢。禁止固定金額、區間、Unit/cc、推給他院比較。

04 / 更新後優化方案

策略四與五

四：高意願先承接，再留資

先記時間意圖（下週/明天/週三晚上），先承接療程困擾，回答探查題先關心不重問；用「專人幫您確認可約時段」作為留資理由。

已覆蓋：test_treatment_time_ready_request_enters_booking_context、test_already_provided_contact_followup_does_not_reask_contact 等。

五：DeepEval 與真實 log 雙軌

① 快速回歸：booking、rag_price、post-op guard、risk guard。
② 完整 DeepEval：繁中品質、Booking Flow、No Quote、Medical Safety。
真實 log 持續抽樣，盯重複追問與百科式長文。
