優化與行動報告

基於 8,637 筆歷史對話數據與系統實測的優化藍圖

專案管理與開發團隊
這是一份針對現有 Facebook 智能客服的優化藍圖報告。

01 / Executive Summary

執行摘要

目前 Facebook 智能客服已成功部署 Multi-Agent ReAct 架構，且 RAG 知識庫與 DeepEval 評估框架已建立基準線 (Baseline)。

現有瓶頸

在「初期流量承接」、「價格詢問委婉轉移」與「轉 LINE 預約」存在斷層，導致 DeepEval 繁中品質等項目未達標。

解決思路

依據「數據佐證問題 ➔ 系統架構優化」邏輯，進行針對性的流程重建與安全攔截防線佈署。

02 / 核心痛點

痛點一：初期流失率高

快速回覆按鈕未能有效推進對話

  • 數據佐證：89.4% 客戶來自廣告，最大流失點為「快速回覆後無後續」(28.06%) 與「客服開場後無回覆」(24.83%)。17.2% 客戶僅點擊按鈕就離開。
  • 系統現狀：首輪回覆較為死板，往往是泛用型問候或直接丟出冗長模板。DeepEval 繁中品質全數 fail。

02 / 核心痛點

價格與私域轉化的挑戰

痛點二：價格詢問應對

約 28.5% 問價。醫美價格因人而異，目前專案擬取消自動報價。系統若生硬拒絕或強制要資料，極易引發反感與流失。

痛點三：轉 LINE 摩擦大

漏斗顯示 57.2% 轉接 LINE，但最終僅 29.4% 預約。系統常在未提供足夠價值前過早要求聯絡方式，導致 16.86% 在此流失。

02 / 核心痛點

海外需求與客訴風險

痛點四：海外客群未處理

國際客戶佔 2.4%，多為高客單價的三層次拉皮客群，需求集中在停留天數、匯率。目前缺乏判斷邏輯。

痛點五：客訴與術後情緒

歷史對話中含 [negative_complaints] 標籤有 26 筆 (0.3%)。佔比雖低但公關與法律風險極大。系統目前缺乏強制轉真人機制。

03 / 優化方案

策略一：Symptom-driven Routing

重構 Orchestrator Agent 的開場與路由邏輯。

  • 行動：放棄「療程導向」按鈕，改為「痛點/決策導向」。
  • 實作：
    • FB 選單改為「我想判斷適合哪一種」、「想了解恢復期與痛感」。
    • 收到痛點進場時，先用二選一問題分型（如「在意嘴邊肉還是下顎線？」），不直接丟衛教長文。

03 / 優化方案

策略二：委婉轉移 SOP

取消自動報價，將問價轉化為「引導評估」的契機。

  • 資料層 (RAG)：清理固定價格，改為「價格評估變因說明」（如手術範圍、鬆弛程度）。
  • 邏輯層 (SalesAgent)：強制採取委婉轉移格式：
    同理與承認問題 ➔ 說明影響費用的變因 ➔ 問一個狀態評估題 ➔ 邀請傳照片或門診評估

03 / 優化方案

策略三：價值優先的聯絡收集

優化 Booking Flow，延遲索取 LINE ID 時機。

  • 實作方式：修改狀態機，在要求個資前，必須先完成「線上照片初評引導」或「醫師專長/案例分享」。
  • 防呆機制：嚴格落實 Contact Bundle，已提供姓名/電話的客戶，系統絕對不可再次鬼打牆詢問或顯示「未提供」。

03 / 優化方案

策略四與五：跨境工作流與安全防線

跨境醫療 (Cross-border)

偵測「馬幣、飛機、馬來西亞」等字眼，自動觸發跨境 SOP：提供視訊選項、預估停留天數，優先轉交高階人工客服。

Medical Safety Guard

加入 RiskGuard 中間件偵測「毀容、失敗、媒體、痛到無法睡」等字。一旦觸發即強制暫停 AI (Bot Pause) 並通知主管火速接手。

04 / 預期效益

Expected Outcomes & KPIs

評估門檻通過

  • 繁中品質平均分提升至 > 0.85
  • RAG 測試 Retrieval Relevancy > 0.90（準確提供變因）

轉換率提升

  • 快速回覆後無後續流失率下降 10%
  • 問價到線上評估承接率從 20.77% 升至 35%

05 / 總結與下一步

Action Items

核心挑戰在於「業務邏輯的細膩度」與「對客戶心理的同理心」。

  • [Data/RAG] 匯入報價變因文件至 ChromaDB。
  • [Agent/Prompt] 調整 Orchestrator / Sales System Prompt。
  • [Testing] 同步部署修正代碼，執行 DeepEval regression 確保綠燈通過。
  • [Product] 評估導入 Temporal 工作流引擎以管理長時狀態。
